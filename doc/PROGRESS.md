tasks with one of four status markers:

- **[done]** — implemented, tested, and merged.

- **[deferred]** — blocked on an external dependency (hardware, SDK, …).

view) and this document (the tactical status board) are updated together.

## §2 — Security Hardening

### [done]
- **Software-key encryption-at-rest.** `KeyStorage.private_key` for
  asymmetric software keys is now AES-256-GCM ciphertext under a
  per-provider KEK derived via HKDF-SHA256 from `HSM_PIN_KEK`. — Phase 0.2.
- **HSM PIN encryption-at-rest.** Provider auth secrets via
  `auth_secret_ref='db:encrypted'` are wrapped under the master KEK and
  stored in `CryptoProviders.auth_secret_blob`. — Phase 4.
- **Per-provider secret reference + auto-activation toggle.** Resolvers
  `db:encrypted` / `env:` / `vault:` / `operator:prompt`; `auto_activate`
  validated against the resolver kind. — Phase 4 (kms-strategy.md §6–7).
- **`HSM_PIN_KEK` plumbing.** Generated by `setup.sh`, written to `.env`,
  passed to the Docker container via `docker-compose.yml`,
  auto-loaded from `.env` in local-dev mode by `web/services/__init__.py`.
- **End-entity key-escrow regime aligned with the per-provider KEK.**
  `generate_pkcs12(store_key=True)` now KEK-wraps in both branches:
  the no-passphrase path stores `storage_type='Encrypted'` (KEK-wrapped
  plaintext PEM, loadable through `KMS.sign_digest`); the passphrase path
  stores `storage_type='PassphraseEncrypted'` (KEK-wrapped passphrase-
  encrypted PEM, accessible only via `build_pkcs12_for_certificate` with
  the operator-supplied passphrase). The new enum value, the
  idempotent migration that adds it, the `db.kek_wrap_pem` public helper,
  the tightened `_looks_like_pem` (excludes PKCS#8 ENCRYPTED + PKCS#1
  Proc-Type: 4,ENCRYPTED so the migration doesn't double-encrypt
  passphrase-PEM), and the explicit refusal in
  `SoftwareBackend.load_key` together restore single-meaning of
  `'Encrypted'` and give defense-in-depth uniform at-rest encryption.
  Touched: `pypki/db.py`, `pypki/core.py`, `pypki/backends/software.py`.
  Regression coverage in `tests/test_pkcs12_storage_regime.py`
  (8 tests: detection refinement, refusal-with-clear-message on the
  sign path, KEK-wrap round-trip, wrong-passphrase failure mode).

### [pending]
- **First-class secret management for admin/bootstrap credentials.**
  Currently generated by `setup.sh` and stored in `.env` /
  `.setup_credentials`; no integration with Vault / k8s secrets / etc.
- **MFA for the management UI.** `auth_routes.py` issues plain JWTs.
- **Shorter-lived / rotatable JWT tokens.** Token TTL and rotation policy
  not configurable; refresh-token flow not implemented.
- **EST client-certificate authentication.** `est_routes.py` only
  supports Basic Auth via the alias `username` / `password_hash`.

## §3 — HSM / PKCS#11 Support

Cross-reference: [kms-strategy.md](kms-strategy.md) for the design,
[hsm-gap-analysis.md](hsm-gap-analysis.md) for the closed-bug catalogue.

### [done]
- **Crypto provider data model (`CryptoProviders` table).** `kind` ∈
  `software` | `pkcs11`, `auth_secret_ref`, `auth_secret_blob`,
  `auto_activate`, FK from `KeyStorage.provider_id`. — Phase 0.1.
- **`SoftwareBackend` and `PKCS11Backend` siblings under one contract.**
  KMS dispatches via the `CryptoBackend` Protocol; software keys via
  libcrypto, HSM keys via PKCS#11. — Phase 0.3.
- **SoftHSM2 dev environment.** Installed in the Docker image, token
  initialised by `docker-entrypoint.sh`, persistent token volume,
  manual setup documented in [softhsm2-manual.md](softhsm2-manual.md). — Phase 0.4.
- **Gap 1 — RSA-on-HSM mechanism.** `CKM_RSA_PKCS` with prepended
  SHA-256 DigestInfo; signatures verify against `sha256WithRSAEncryption`.
  — Phase 1.
- **Gap 2 — ECDSA-on-HSM branch.** `CKM_ECDSA` over the digest, raw
  `r||s` DER-encoded via `encode_dss_signature`. — Phase 1.
- **Gap 3 — slot resolution by token label.** `slot_label` threaded
  through `PKCS11Backend.open` → `PKCS11Helper.open_session`; matches
  `CK_TOKEN_INFO.label`, errors clearly when label is missing. — Phase 2.
- **Gap 4 — provider abstraction.** Closed by Phase 0 (provider model).
- **Gap 5 — PIN out of MySQL plaintext.** `auth_secret_ref` resolvers,
  `auto_activate` enforced. — Phase 4.
- **Gap 6 — one PKCS#11 session per provider.** `PKCS11Backend` owns the
  session for the lifetime of activation; `KMS.shutdown()` and atexit
  hook close cleanly; reconnect-on-`CKR_SESSION_HANDLE_INVALID`. — Phase 3.
- **Gap 7 — `KMS.load_key` thread-safety.** Double-checked locking around
  the cache; per-backend RLock serialises sign calls. Multi-threaded
  stress test passes. — Phase 3.
- **Gap 8 — operator UX (API + UI).** Provider CRUD, key generate /
  import / delete, mandatory CKA_* attribute set on PKCS#11-generated
  keys, owned-vs-imported delete semantics, audit-log events. — Phase 5.
- **Gap 9 — `hsm_token_id` hex validation.** Validated at API boundary
  (`_validate_cka_id_hex`); typos fail at insert/import not at sign.
  — Phase 6.
- **Gap 10 — `storage_type='Encrypted'` decrypt path.** Closed by
  construction in Phase 0.2 (encrypt-at-rest); legacy plaintext-PEM-in-
  Encrypted detection added in Phase 5b for upgraded deployments.
- **Gap 11 — `Symmetric` storage type.** AES keys persist as
  `storage_type='Symmetric'`; load path rejects with a clear error.
  — Phase 6.
- **Gap 12 — stale OCSP error reference.** `migrate_keys_to_kms.py`
  reference replaced. — Phase 6.
- **`pkcs11_helper.py` slim-down.** Removed eight dead methods,
  `PKCS11_LIB` constant, and the `get_key_by_id` method (moved into
  `PKCS11Backend`). — Phase 6.
- **`KeyTools` HSM-path removal.** `KeyTools` is now purely a
  software-key wrapper; the HSM construction branch and legacy KMS
  fallback are gone. — Phase 6.

### [partial]
- **Test parametrisation across backends.** Software and PKCS#11 KMS
  tests live in separate files exercising the same KMS API. Not literally
  `pytest.mark.parametrize` over backends — both are exercised, but
  individual test cases pick their own. Could tighten to a single
  parametrised matrix.

### [deferred]
- **Fidelity pass — YubiHSM 2 simulator.** Pending Yubico SDK download +

- **Fidelity pass — Thales DPoD.** Pending DPoD trial registration.

## §8 — Functional Improvements

Cross-reference: [roadmap.md §8](roadmap.md). New product-capability
change requests land here as `[pending]` entries.

### [done]
- **Self-signed certificate option in the request flow.** Sentinel
  "— Self-signed (no CA) —" entry at the top of the CA dropdown on
  `certificate_request.html` (`value=0`). Routes
  `/api/certificate/issue-pkcs12` and `/api/certificate/issue` accept
  `ca_id ∈ {null, 0}` as the self-signed sentinel. The CSR path
  rejects self-signed cleanly with a 400 and an actionable message
  (the server doesn't have the CSR's private key, so the freshly
  generated key from server-side keygen is required). The selected
  template still drives subject / SAN / extensions / validity; the
  freshly-generated private key signs the certificate as its own
  issuer. Validity is capped to `template.max_validity` unless that
  is `INFINITE_VALIDITY (-1)`, in which case the cert is unbounded
  (uses GeneralizedTime far-future). New schema column
  `Certificates.is_self_signed BOOLEAN NOT NULL DEFAULT FALSE`
  added with an idempotent migration; `db.insert_certificate`
  accepts the flag. The KMS path is unchanged — the cert key still
  goes through the default software provider. Touched:
  `pypki/db.py`, `pypki/core.py`, `web/services/api_adapters.py`,
  `web/routes/main_routes.py`, `web/templates/certificate_request.html`.

## How to use this document

- **Triaging new work:** before starting, find the matching `[pending]`
  task here (or add one). When complete, flip to `[done]` with a
  one-line note about which phase / commit / PR landed it.
- **Reviewing the project state:** the at-a-glance table is a
  reasonable executive summary. The per-section breakdown is the
  honest operator-facing read.
- **Adding new requirements:** add the task here as `[pending]` and
  link it from the matching roadmap.md bullet. Don't update only
  one of the two — they drift apart fast otherwise.

When in doubt about whether something is `[done]` or `[partial]`,
prefer `[partial]` and spell out the gap. Honest tracking beats green
checkmarks.
